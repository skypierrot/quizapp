# 기술 구현(업데이트) 예정 목록

## 파일 개요

이 문서는 현재 프로젝트 진행 상황, 고객 제안서 및 기술 구현 완료 목록을 바탕으로, 향후 구현하거나 개선할 주요 기술 기능 및 항목을 상세하게 정리한 목록입니다. 프로젝트의 방향성을 명확히 하고 개발 우선순위를 결정하는 데 활용됩니다.

## 주요 업데이트 예정 목록

### 1. 핵심 기능 구현 및 개선

*   **Clerk 인증 연동 활성화 및 강화:**
    *   **목표:** Clerk을 이용한 안전한 사용자 인증 및 세션 관리 활성화.
    *   **개발 방향 및 접근 방식:**
        *   `middleware.ts` 수정:
            *   `clerkMiddleware` 함수를 기본 export로 사용.
            *   `createRouteMatcher`를 사용하여 `['/sign-in(.*)', '/sign-up(.*)', '/api/webhooks/clerk']` 등 공개 경로 정의.
            *   `auth().protect()`를 사용하여 비공개 경로(예: `/manage/**`, `/learn/**`, `/test/**`, `/profile/**`, 대부분의 `/api/**` 경로) 접근 제어 적용.
            *   `config` 객체의 `matcher`를 조정하여 미들웨어가 필요한 모든 경로(정적 파일, `_next` 제외)에 적용되도록 확인.
        *   Clerk 사용자 정보와 `users` 테이블 동기화:
            *   **방법 1 (Webhook):** Clerk Dashboard에서 User 생성/수정 이벤트에 대한 Webhook 설정. Webhook 요청을 받을 전용 API 라우트 핸들러 (`/api/webhooks/clerk`) 생성. 해당 핸들러에서 이벤트 타입에 따라 Drizzle ORM을 사용하여 `users` 테이블에 사용자 정보(`clerkId`, `email`, `name`, `image` 등)를 `INSERT` 또는 `UPDATE` 수행. (권장: 서버 간 통신으로 안정적)
            *   **방법 2 (최초 로그인 시):** 로그인 후 사용자가 접근하는 특정 페이지(예: 대시보드) 또는 API 요청 시, 해당 사용자의 `clerkId`가 `users` 테이블에 없는 경우 `clerkClient().users.getUser()`로 Clerk에서 정보를 가져와 `users` 테이블에 삽입하는 로직 추가. (구현은 간단하나 매번 체크 필요)
        *   (선택적) 사용자 역할 기반 접근 제어:
            *   Clerk Metadata 활용: 사용자 생성 시 또는 관리자 페이지에서 Clerk 사용자의 Public/Private Metadata에 역할 정보 (예: `{ "role": "admin" }`) 저장.
            *   API 핸들러 또는 페이지 컴포넌트에서 `auth().sessionClaims?.metadata.role` 또는 `auth().protect({ role: 'org:admin' })` (Organization 사용 시) 등을 사용하여 역할 확인 및 접근 제어 구현.

*   **게시판 기능 구현 (`/community/...` 경로):** (신규)
    *   **목표:** 공지사항 및 사용자 간 학습 정보 공유를 위한 게시판 기능 제공.
    *   **개발 방향 및 접근 방식:**
        *   **데이터 모델 정의 (`db/schema/community.ts`):**
            *   `posts` 테이블: `id`(uuid, pk), `userId`(uuid, fk->users.id), `title`(text), `content`(text), `type`('notice'|'forum'), `createdAt`, `updatedAt`.
            *   `comments` 테이블: `id`(uuid, pk), `userId`(uuid, fk->users.id), `postId`(uuid, fk->posts.id), `content`(text), `createdAt`, `updatedAt`.
            *   `users` 테이블과의 관계 설정 (작성자 정보 표시용).
        *   **API 구현 (`app/api/community/...`):**
            *   `GET /api/community/posts?type={type}`: 게시글 목록 조회 (페이지네이션, 타입 필터링).
            *   `POST /api/community/posts`: 새 게시글 작성 (Clerk 인증 필요).
            *   `GET /api/community/posts/[postId]`: 특정 게시글 상세 조회.
            *   `PUT /api/community/posts/[postId]`: 게시글 수정 (작성자 또는 관리자 권한 확인 필요).
            *   `DELETE /api/community/posts/[postId]`: 게시글 삭제 (작성자 또는 관리자 권한 확인 필요).
            *   `GET /api/community/posts/[postId]/comments`: 특정 게시글의 댓글 목록 조회.
            *   `POST /api/community/posts/[postId]/comments`: 새 댓글 작성 (Clerk 인증 필요).
            *   `DELETE /api/community/comments/[commentId]`: 댓글 삭제 (작성자 또는 관리자 권한 확인 필요).
            *   각 핸들러 시작 시 `auth()`로 사용자 인증 및 권한 확인 로직 포함. Zod로 입력값 유효성 검사.
        *   **UI 구현 (`app/community/...`):**
            *   페이지 구조: `/community/notices`, `/community/notices/[postId]`, `/community/forum`, `/community/forum/[postId]`, `/community/new` (또는 각 게시판 내 작성 버튼).
            *   UI 컴포넌트: `PostList` (ShadCN `Table` 또는 `Card` 리스트), `PostView` (게시글 내용, 작성자 정보, 댓글 목록 `CommentSection` 포함), `PostForm` (ShadCN `Input`, `Textarea` 사용. Rich Text Editor 필요시 Tiptap 또는 React Quill 라이브러리 검토 및 연동).
            *   상태 관리: React Query 또는 SWR 도입하여 API 데이터 fetching, caching, mutation 관리 고려 (데이터 재검증, 로딩/에러 상태 처리 용이). 없다면 `useState`, `useEffect` 사용.

*   **문제 관리 기능 개선 (`/manage/...`):**
    *   **목표:** 문제 생성/관리 편의성 및 데이터 정합성 향상.
    *   **개발 방향 및 접근 방식:**
        *   `PasteForm` API 연동 수정:
            *   `/api/questions/batch` API 핸들러에서 `request.formData()`를 사용하여 `FormData` 파싱. 텍스트 데이터와 이미지 파일 분리 처리.
            *   이미지 파일은 `/api/images/upload` 와 유사하게 임시 저장 후, 문제 데이터와 함께 DB 저장 시 `/api/images/finalize` 로직 활용 또는 통합 처리. (트랜잭션 처리 중요)
        *   태그 관리 시스템 개선:
            *   태그 자동완성 API (`GET /api/tags?q={query}`): `questions` 테이블의 `tags` 필드를 분석하여 고유 태그 목록을 제공하는 API 구현.
            *   `ManualForm`의 태그 입력 필드에 ShadCN `Combobox` 또는 유사 컴포넌트를 사용하여 자동완성 기능 구현.
            *   `/manage/questions/list` 페이지에 태그 필터링 UI 추가 (예: 다중 선택 가능한 `Checkbox` 또는 `Select` 컴포넌트 사용) 및 `fetchQuestions` API 호출 시 `tags` 쿼리 파라미터 전송 로직 수정.
        *   DB 스키마 관계 설정 (`db/schema/images.ts`, `db/schema/questions.ts`):
            *   Drizzle 스키마 정의 시 `images.questionId` 컬럼에 `references(() => questions.id)` 추가하여 외래 키 제약 조건 설정. (삭제 옵션 `onDelete: 'set null'` 또는 `onDelete: 'cascade'` 고려)
        *   문제 삭제 로직 개선:
            *   `questions` 테이블에 `status` 컬럼 추가 (`pgEnum('question_status', ['active', 'deleted'])` 사용) 또는 `deletedAt` (timestamp, nullable) 컬럼 추가.
            *   `DELETE /api/questions/[questionId]` API 핸들러 로직을 실제 행 삭제 대신 `status` 업데이트 또는 `deletedAt` 타임스탬프 설정으로 변경.
            *   `GET /api/questions` 등 조회 API에서 `status = 'active'` 또는 `deletedAt IS NULL` 조건 기본 적용.

*   **모의고사 기능 구체화 (`/test/...`):**
    *   **목표:** 실제 시험과 유사한 환경의 모의고사 응시 및 결과 확인 기능 제공.
    *   **개발 방향 및 접근 방식:**
        *   **데이터 모델 정의 (`db/schema/index.ts`, `db/schema/exams.ts`):**
            *   `exams` 테이블: `id`(uuid, pk), `name`(text, notNull), `description`(text). (시험 자체 정보)
            *   `examResults` 테이블 상세화: `id`(uuid, pk), `userId`(uuid, fk->users.id), `examId`(uuid, fk->exams.id), `examInstanceInfo`(jsonb, 응시한 시험 회차 정보: 년도, 회차 등), `score`(integer), `totalQuestions`(integer), `correctAnswers`(integer), `completedAt`(timestamp), `timeTakenSeconds`(integer), `answers`(jsonb, `[{ questionId: string, selectedOptionIndex: number | null, isCorrect: boolean }]` 구조).
        *   **API 구현 (`app/api/test/...`):**
            *   `POST /api/test/sessions`: 모의고사 시작 요청. 요청 바디에 시험 회차 정보(또는 examId). 해당 회차 문제 목록 조회, `examResults` 레코드 초기 생성(상태 'in_progress'), 세션 ID(resultId) 반환.
            *   `GET /api/test/sessions/[sessionId]/questions`: 특정 세션의 문제 목록 반환 (보안 고려: 이미 시작된 세션의 사용자만 접근 가능해야 함).
            *   `POST /api/test/sessions/[sessionId]/submit`: 답안 제출 요청. 요청 바디에 사용자의 답안 배열 (`{ questionId, selectedOptionIndex }[]`).
                *   서버에서 해당 문제들의 정답 정보 조회하여 채점 수행.
                *   `examResults` 레코드 업데이트 (score, correctAnswers, completedAt, timeTakenSeconds, answers 필드 채우기, 상태 'completed').
            *   `GET /api/test/results/[resultId]`: 특정 모의고사 결과 상세 조회 (결과 소유자만 접근 가능해야 함).
        *   **UI 구현 (`app/test/session/[sessionId]`, `app/test/result/[resultId]`):**
            *   **모의고사 진행 페이지:**
                *   타이머: `useState`, `useEffect` 사용하여 남은 시간 관리 및 1초마다 업데이트. 시간 종료 시 자동 제출 로직.
                *   문제 표시: `StudyPage` 컴포넌트 재사용 또는 모의고사 전용 문제 표시 컴포넌트 개발. (선택지 클릭 시 사용자 답안 상태 저장 필요)
                *   네비게이션: 문제 목록 사이드바 또는 하단 버튼으로 문제 간 이동 지원.
                *   상태 관리: 현재 문제 인덱스, 사용자 답안 배열 등을 `useState` 또는 `useReducer`로 관리. `localStorage` 활용하여 페이지 새로고침 시 상태 유지 고려.
            *   **결과 페이지:**
                *   점수, 정답/오답 개수, 소요 시간 등 요약 정보 표시.
                *   응시한 문제 목록과 사용자 선택 답안, 정답 비교 표시. (틀린 문제 하이라이트)
                *   (선택적) 차트 라이브러리(예: Recharts) 사용하여 문항별 정답률 등 시각화.

*   **오답 노트 기능 구현 (`/learn/notes` 또는 `/profile/notes` 경로):**
    *   **목표:** 모의고사 또는 학습 중 틀린 문제를 모아보고 다시 풀어보는 기능 제공.
    *   **개발 방향 및 접근 방식:**
        *   **데이터 모델:** `examResults` 테이블의 `answers` 필드(오답 정보 포함)를 활용하거나, 별도 `omittedNotes` 테이블(`userId`, `questionId`, `createdAt`, `lastReviewedAt`) 관리 고려. (후자가 복습 기능 확장에 유리)
        *   **API 구현 (`app/api/notes/...`):**
            *   `GET /api/notes`: 현재 사용자의 오답 목록 조회. `examResults` 테이블을 조회하여 `answers` 필드에서 오답(`isCorrect: false`)인 `questionId` 목록 추출. 또는 `omittedNotes` 테이블 조회. (필터링 파라미터 추가: 과목, 날짜 등)
            *   `GET /api/notes/questions?ids={questionId1},{questionId2}`: 오답 다시 풀기를 위해 특정 문제 ID 목록에 해당하는 문제 데이터 조회.
        *   **UI 구현 (`app/learn/notes`):**
            *   오답 목록 필터링 UI (과목, 날짜 등).
            *   오답 목록 표시 (`Table` 또는 `Card` 리스트).
            *   '다시 풀기' 버튼 클릭 시 선택된 문제들을 학습 페이지와 유사한 인터페이스로 제공. 정답 확인 기능 포함.
            *   (선택적) 오답 목록에서 특정 문제 제거 기능.

*   **주제별/개념별 심층 학습 (`/learn/topics`):** (신규 제안)
    *   **목표:** 사용자가 특정 과목 내 세부 주제나 핵심 개념을 선택하여 집중적으로 학습할 수 있도록 지원.
    *   **개발 방향 및 접근 방식:**
        *   문제에 연결된 핵심 개념/이론 요약 제공: 문제 데이터 스키마에 관련 개념 ID 또는 텍스트 필드 추가. 문제 풀이 화면에서 해당 정보 표시/링크.
        *   주제별 문제 묶음 풀이: `questions` 테이블의 태그 또는 별도 `topic` 테이블과의 관계를 이용하여 특정 주제 관련 문제 필터링 API (`GET /api/questions?topic={topicId}`) 구현. UI에서 주제 선택 후 해당 문제 목록 로드 및 학습 인터페이스 제공.

*   **맞춤형 약점 공략 문제 추천 (`/learn/recommended`):** (신규 제안)
    *   **목표:** 사용자의 학습 데이터(오답률, 풀이 시간 등)를 분석하여 취약한 유형의 문제를 시스템이 추천해주어 효율적인 학습 유도.
    *   **개발 방향 및 접근 방식:**
        *   추천 로직 API (`GET /api/recommendations/questions`): `examResults` 테이블 분석 (사용자 ID 기반, 오답률 높은 태그/유형 식별)하여 추천 문제 ID 목록 반환 로직 구현.
        *   UI 구현 (`app/learn/recommended`): API 호출하여 추천 문제 목록 표시. 각 문제 클릭 시 학습 인터페이스로 연결.

*   **학습 계획 및 진도 관리 도우미 (`/learn/planner` 또는 대시보드 연동):** (신규 제안)
    *   **목표:** 사용자가 스스로 학습 목표를 설정하고 진행 상황을 관리하며 꾸준한 학습 습관 형성 지원.
    *   **개발 방향 및 접근 방식:**
        *   데이터 모델 (`learningPlans` 테이블 등): `userId`, `goalType`('count'|'duration'), `goalValue`, `startDate`, `endDate`, `status`.
        *   API 구현: 학습 목표 생성/수정/삭제 API, 학습 활동 기록 API (예: 문제 풀이 시 자동 기록), 목표 대비 진도 조회 API.
        *   UI 구현: 목표 설정 폼, 학습 캘린더(예: `react-calendar` + 커스텀), 진행률 시각화(ShadCN `Progress` 등).

*   **핵심 용어/개념 사전 (`/learn/glossary`):** (신규 제안)
    *   **목표:** 시험 관련 전문 용어 및 필수 개념에 대한 쉬운 접근 및 학습 지원.
    *   **개발 방향 및 접근 방식:**
        *   데이터 모델 (`glossaryTerms` 테이블): `id`, `term`, `definition`, `subject`, `relatedQuestionIds`(jsonb 또는 별도 연결 테이블).
        *   API 구현: 용어 검색/목록 조회 API (`GET /api/glossary?q={query}`).
        *   UI 구현: 검색 바, 용어 목록 표시, 상세 설명 페이지. 문제 풀이 중 용어 클릭 시 팝업 또는 링크로 사전 연결.

*   **문제 풀이 중 메모 및 하이라이트 기능:** (신규 제안)
    *   **목표:** 문제 풀이 중 개인적인 노트 기록 및 복습 효율 증대.
    *   **개발 방향 및 접근 방식:**
        *   데이터 모델 (`userNotes` 테이블): `userId`, `questionId`, `noteContent`(text), `highlightRanges`(jsonb), `createdAt`, `updatedAt`.
        *   API 구현: 특정 문제에 대한 노트/하이라이트 저장/조회/삭제 API.
        *   UI 구현: 문제 풀이 화면(학습, 오답 노트 등)에 메모/하이라이트 도구 추가 (예: `react-quill` 활용 또는 직접 구현). 저장된 내용 로드 및 표시 로직. (선택적) 노트 모아보기 페이지.

*   **모바일 접근성 강화:**
    *   **목표:** 모바일 환경에서도 모든 기능을 불편함 없이 사용할 수 있도록 보장.

        *   모의고사 진행 화면: 작은 화면에서는 명확하게 한 문제와 선택지만 표시되도록 레이아웃 조정. 버튼 크기 및 터치 영역 확보.

        *   모바일 브라우저 개발자 도구를 이용한 지속적인 테스트 및 디버깅.

    *   **목표:** 애플리케이션 응답 속도 향상 및 서버 부하 감소.
    *   **개발 방향 및 접근 방식:**
